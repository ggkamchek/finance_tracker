# 01. Анализ предметной области

## 1. Контекст и цель проекта

`finance_tracker` - локальное консольное приложение для персонального учета доходов и расходов.  
Ключевая цель - обеспечить пользователю быстрый и понятный контроль личных финансов без внешних сервисов и сетевой инфраструктуры.

## 2. Границы предметной области

### 2.1 Входит в область

- регистрация транзакций (доход/расход);
- хранение атрибутов транзакции (сумма, категория, дата, описание);
- операции с данными: добавление, просмотр, редактирование, удаление;
- фильтрация списка по категории;
- расчет текущего баланса;
- загрузка и сохранение состояния в `JSON`.

### 2.2 Не входит в область

- интеграции с банками, платежными системами и API;
- многопользовательский режим и разграничение прав доступа;
- аналитика с графиками, прогнозами и дашбордами;
- web/mobile интерфейсы и облачная синхронизация.

## 3. Участники и интересанты

- **Пользователь (основной актер):** ведет учет операций и контролирует баланс.
- **Разработчик/практикант:** развивает функциональность, поддерживает тесты и документацию.
- **Преподаватель/ревьюер:** оценивает полноту реализации, качество артефактов и корректность тестирования.

## 4. Ключевые сущности и их атрибуты

### 4.1 Сущность `Transaction`

- `ID` - уникальный целочисленный идентификатор;
- `Type` - тип операции: `income` или `expense`;
- `Amount` - сумма операции (`> 0`);
- `Category` - категория из фиксированного списка;
- `Date` - дата в формате `DD.MM.YYYY`;
- `Description` - произвольный текст.

### 4.2 Сущность `JSONStorage`

Инфраструктурный компонент, который:
- загружает транзакции из `data/transactions.json`;
- сохраняет текущее состояние списка транзакций в файл;
- сообщает ошибку при некорректном формате данных.

## 5. Бизнес-правила

- сумма операции должна быть строго положительной;
- тип операции ограничен двумя значениями (`income`, `expense`);
- категория должна соответствовать словарю приложения;
- формат даты единообразный (`DD.MM.YYYY`);
- при отсутствии файла хранилища система начинает работу с пустого набора данных.

## 6. Основные бизнес-процессы

1. **Ввод операции:** пользователь вводит параметры -> сервис валидирует -> операция добавляется.
2. **Корректировка:** поиск по `ID` -> редактирование/удаление -> повторная валидация.
3. **Контроль расходов:** фильтрация по категории и расчет итогового баланса.
4. **Персистентность:** чтение данных при старте и запись при сохранении/выходе.

## 7. Ограничения, допущения, риски

### Ограничения
- один пользователь и один локальный файл данных;
- отсутствие транзакционности и блокировок на уровне БД.

### Допущения
- пользователь корректно интерпретирует CLI-подсказки;
- проект используется в учебных целях, без требований высокой нагрузки.

### Риски
- потеря последних изменений при аварийном завершении до сохранения;
- ручное повреждение `JSON`-файла, приводящее к ошибке чтения;
- человеческий фактор при вводе данных (снижается встроенной валидацией).
