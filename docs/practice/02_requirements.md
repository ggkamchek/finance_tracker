# 02. Требования к системе

## 1. Функциональные требования (FR)

### FR-01. Добавление транзакции
- Система должна создавать транзакцию на основе введенных полей: `type`, `amount`, `category`, `date`, `description`.
- Система должна присваивать новой записи уникальный `ID`.

### FR-02. Просмотр транзакций
- Система должна выводить полный список транзакций в человекочитаемом формате.

### FR-03. Редактирование по `ID`
- Система должна позволять изменять существующую транзакцию по ее идентификатору.
- При обновлении должны применяться те же валидаторы, что и при создании.

### FR-04. Удаление по `ID`
- Система должна удалять транзакцию по идентификатору.
- При неверном `ID` система должна возвращать информативное сообщение об ошибке.

### FR-05. Фильтрация по категории
- Система должна отображать только записи выбранной категории.

### FR-06. Расчет баланса
- Система должна вычислять итог: `sum(income) - sum(expense)`.

### FR-07. Сохранение данных
- Система должна сохранять актуальный список транзакций в `data/transactions.json`.

### FR-08. Загрузка данных при старте
- Система должна загружать данные из JSON-файла при запуске приложения.
- При отсутствии файла система должна стартовать с пустым списком, без аварийного завершения.

### FR-09. Справка по командам
- Система должна предоставлять команду помощи (`помощь`/`help`) со списком доступных операций.

## 2. Нефункциональные требования (NFR)

### NFR-01. Надежность
- Ошибки чтения/записи и парсинга данных должны обрабатываться контролируемо.
- Для некорректного JSON должна возвращаться диагностируемая ошибка.

### NFR-02. Производительность
- Для учебного объема до 10 000 записей операции CLI должны выполняться без заметных задержек.

### NFR-03. Удобство использования
- Команды и сообщения должны быть понятны пользователю без внешней документации.

### NFR-04. Портативность
- Приложение должно запускаться в окружениях, поддерживаемых runtime Go.

### NFR-05. Поддерживаемость
- Код должен быть разделен по ответственности: CLI, бизнес-логика, хранение данных.

### NFR-06. Тестируемость
- Критические сценарии хранения данных должны быть покрыты автотестами.

## 3. Правила валидации

- `amount > 0`;
- `type ∈ {income, expense}`;
- `category` принадлежит фиксированному набору категорий;
- `date` соответствует формату `DD.MM.YYYY`.

## 4. Критерии приемки

- реализованы все FR-01..FR-09;
- ключевые ошибки обрабатываются согласно NFR-01;
- сценарии хранения данных подтверждены unit-тестами;
- команда `go test ./...` завершается успешно;
- README содержит ссылки на комплект артефактов практики в `docs/practice`.
