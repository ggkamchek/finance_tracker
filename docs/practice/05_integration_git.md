# 05. Интеграция и стратегия Git

## 1. Цель интеграции

Обеспечить безопасную поставку изменений в проект `finance_tracker`: прозрачная история, быстрый код-ревью и воспроизводимая проверка качества.

## 2. Пошаговая схема интеграции

1. Синхронизировать локальную базу:
   - `git checkout main`
   - `git pull`
2. Создать изолированную рабочую ветку:
   - `git checkout -b feature/practice-docs-tests`
3. Выполнять изменения малыми блоками:
   - документация (`docs/practice/01..07`);
   - тесты (`internal/storage/json_storage_test.go`);
   - минимальные правки `README.md`.
4. Перед каждым коммитом выполнять локальную проверку:
   - `go test ./...`
5. Отправить ветку:
   - `git push -u origin feature/practice-docs-tests`
6. Создать Pull Request с понятным описанием цели и результатов тестов.
7. Исправить замечания ревью и дождаться зеленого CI.
8. Выполнить merge в `main`.

## 3. Коммит-стратегия

### Принципы
- один коммит = одна логическая задача;
- сообщение коммита объясняет, зачем внесено изменение;
- документация и код не смешиваются без необходимости.

### Рекомендуемая последовательность коммитов
1. `docs: add russian practice artifacts 01-07`
2. `test: cover json storage edge scenarios`
3. `docs: link practice artifacts in readme`

## 4. Требования к Pull Request

- заголовок отражает результат, а не только действие;
- описание включает:
  - контекст задачи;
  - список измененных файлов;
  - вывод `go test ./...`;
  - известные ограничения/риски;
- объем PR должен позволять ревью за один проход.

## 5. Definition of Done (интеграция)

- все необходимые изменения внесены в отдельной ветке;
- локальные и CI-тесты проходят;
- документация синхронизирована с текущим состоянием проекта;
- PR одобрен и влит без конфликтов.
