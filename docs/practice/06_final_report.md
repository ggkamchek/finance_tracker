# 06. Черновик итогового отчета по практике

## 1. Титульный блок (черновик)

- Проект: `finance_tracker`
- Тип системы: консольное приложение для персонального учета финансов
- Технологии: Go, стандартная библиотека, JSON-файл для хранения
- Формат выполнения: индивидуальная учебная практика

## 2. Цель и задачи практики

### Цель
Освоить полный цикл разработки прикладного ПО: анализ -> требования -> проектирование -> реализация -> тестирование -> интеграция.

### Основные задачи
- исследовать предметную область учета доходов и расходов;
- формализовать функциональные и нефункциональные требования;
- описать архитектуру и процессы через диаграммы;
- проверить ключевые сценарии ручными и автоматизированными тестами;
- подготовить комплект проектной документации.

## 3. Краткая характеристика реализованного решения

`finance_tracker` поддерживает операции:
- добавление транзакций;
- просмотр, редактирование и удаление;
- фильтрацию по категориям;
- расчет текущего баланса;
- сохранение/загрузку состояния в `data/transactions.json`.

Архитектура разделена на три слоя:
- `main.go` - CLI и взаимодействие с пользователем;
- `internal/transactions` - бизнес-правила и операции над сущностями;
- `internal/storage` - сериализация/десериализация данных.

## 4. Выполненные артефакты практики

- `01_domain_analysis.md` - анализ предметной области;
- `02_requirements.md` - требования и критерии приемки;
- `03_uml_bpmn.md` - диаграммы use case/activity/component/sequence;
- `04_test_cases.md` - ручные сценарии тестирования и критерии завершения;
- `05_integration_git.md` - процесс интеграции и коммит-стратегия;
- `07_practice_diary.md` - заполненный дневник по дням.

## 5. Тестирование и результаты

### Ручное тестирование
Подготовлен набор кейсов для основных пользовательских потоков: запуск, CRUD, фильтрация, баланс, сохранение и обработка ошибок.

### Автоматизированное тестирование
Для слоя `JSONStorage` покрыты критические сценарии:
- отсутствие файла данных (`load missing file`);
- корректный roundtrip сохранения/загрузки (`save+load roundtrip`);
- ошибка при невалидном JSON (`invalid json error`).

Контроль запуска: `go test ./...`.

## 6. Трудности и способы их решения

- **Риск поврежденного JSON:** обработка ошибок парсинга и явная диагностика.
- **Риск некорректного ввода:** валидация обязательных полей и форматов.
- **Риск несогласованной документации:** синхронизация README с артефактами `docs/practice`.

## 7. Итог и перспективы

Цели практики достигнуты: сформирован комплект документации, подтверждена корректность ключевых сценариев и повышена надежность слоя хранения.

Перспективы развития:
- отчеты по периодам и категориям;
- импорт/экспорт данных;
- расширенный интерфейс (TUI/Web) при сохранении текущей бизнес-логики.
